01 · Architecture

A Cloudflare-native Next.js app, with Clerk for identity

Clerk owns who you are. The app's own database owns what you can do. Everything else — requests, receipts, approvals — lives on Cloudflare.

Identity

Clerk

Hosted sign-in / sign-up. Issues the session. App links it via User.clerkUserId.

Authorization

App DB (D1 + Drizzle)

Source of truth for global role, scoped admin roles, team memberships, profile & payout.

Edge gate

proxy.ts

Clerk middleware. Coarse auth only — protects every route except the public ones.

Storage

R2

Receipts as r2://<key>, private + auth-gated download proxy.

Async

Queues + AI

Receipt parsing offloaded to a consumer Worker; Gemini extracts line items.

Per-page

Server guards

Fine authorization lives in each page/route via auth() + canManage* predicates.

Key mental model: the Edge middleware only checks "are you logged in?". The real "are you allowed?" decision is re-made server-side on every page and API route from the app DB's role + scope + membership data. Clerk never decides authorization.

02 · End-to-end

One reimbursement, creation → payment

The lifecycle threads through identity, roles, approvals, and notifications — each owned by a different subsystem.

  1. Identity & provisioning. First authenticated request lazily creates an app User row keyed on clerkUserId (role forced to USER). That row — not Clerk — drives everything downstream.
  2. Membership gate. A plain USER with no approved team membership sees an empty navbar and cannot create a request. The originator is, in practice, an approved Parent/Mentor or Coach.
  3. Draft & submit. Owner drafts, uploads receipts (parsed by Gemini), edits line items, then submits. transitionRequestStatus atomically writes status + an approvalAction + an audit log and recomputes the total.
  4. Coach review. The coach (or an admin acting at the coach stage) approves/rejects. Approval fans out a "ready for admin review" notification to the scope-resolved admin set.
  5. Admin review & payment. A scoped/super admin sees it in the inbox, approves, then marks it PAID. Payout (Zelle) happens out-of-band; PAID is terminal.
  6. Reopen loop. A rejected request can be reopened to DRAFT — but silently (no notification, no cache bust).

The seam that matters: the User row's email is the single delivery key for all notifications and the basis for "don't notify the actor of their own action." Identity drift in Clerk silently breaks both. (More in §06–07.)

05 · Approvals

The request state machine

Transitions are declared in status.ts and enforced centrally by transitionRequestStatus() — every change writes status + approval action + audit log atomically.

DRAFT→ SUBMITTED→ COACH_APPROVED→ ADMIN_APPROVED→ PAID
SUBMITTED→ COACH_REJECTED⟲ DRAFT COACH_APPROVED→ ADMIN_REJECTED⟲ DRAFT

Dual fast-path

If an admin or the team's own coach submits, SUBMITTED → COACH_APPROVED auto-chains, skipping coach review.

Authorization

Coach stage: any COACH on the team or a reimbursement admin. Admin stage: reimbursement admin only.

PAID is terminal

No transition out — no reopen, no reversal. An erroneous "mark paid" can only be fixed in the DB.

Two structural gaps: (1) routes check role but not current status before transitioning — an illegal transition surfaces as a raw 500 from assertTransition instead of a clean 409. (2) The total is recomputed live on every transition, so the amount marked PAID may differ from what the admin approved if line items changed in between.

06 · Notifications

How notifications percolate

In-app only — no email / SMS / push anywhere. A single sendNotification() resolves recipient emails → user rows and inserts one row per recipient. The bell polls every 30s, but only while open.

The chain only flows up, never back down. Admin-stage decisions notify only the creator — the coach who approved and other admins get no closing signal. Reopen notifies no one. So no single actor reliably sees a request's full lifecycle.

06 · Notifications

Recipient resolution is scope-aware

Admin recipient query

For a request's {districtId, schoolId, programId}, getAdminReviewRecipientEmails ORs three matches:

  • Every global SUPER_ADMIN
  • SCHOOL_ADMIN matching the school, or the district when school is null (district-wide)
  • PROGRAM_ADMIN matching both school and program

Returns deduped, non-null emails; the acting admin is filtered out by email !== actor.email.

The fragile link: email

Delivery is keyed entirely on users.email. sendNotification looks up recipients with inArray(users.email, …) and silently returns if none match — no log, no fallback.

If a coach/admin changes their email in Clerk, the app DB keeps the stale value (no sync). Their notifications stop arriving and the actor-self-exclusion comparison can misfire — all from one drifted field.

07 · Profile & auth

Clerk owns identity; the app owns the rest

On first authenticated request, the app lazily provisions a User row. Name & email are copied from Clerk once and never re-synced.

  1. Clerk middleware requires a session; app calls auth() → gets clerkUserId.
  2. Look up User by clerkUserId; if found, return verbatim.
  3. Otherwise read the Clerk profile, lowercase the email, and insert … onConflictDoUpdate(target: email) — this adopts a pre-seeded row by email and backfills clerkUserId.
  4. Role is always forced to USER; elevation only happens via seed/DB.

App-owned, editable on /profile

  • Mailing address
  • Zelle payout type + value
  • Role, scopes, team memberships, onboarding

Clerk-owned, not editable in app

  • Name & email (copied once, drift forever)
  • Password / auth factors
  • The profile page can't even display or correct them
07 · Profile & auth

Three high-severity auth gaps

HIGH Policy consent never recorded for real users

policyAcceptedAt / policyVersion are written only by the seed script. The Clerk provisioning path never sets them, so every real user has null — yet the profile UI hardcodes "Policy accepted during registration." A UX lie and a compliance gap with no auditable consent.

src/auth.ts · src/components/profile/profile-form.tsx

HIGH No Clerk → DB sync (name / email drift)

No webhook, no re-read on later logins. Change your email in Clerk and the app keeps the old one indefinitely — corrupting display name, notification targeting, and every email-based match.

src/auth.ts

HIGH Email-match account adoption is a takeover vector

Provisioning binds any Clerk login to an existing row by email, without verifying the email is verified. A Clerk account matching a seeded SUPER_ADMIN email inherits that privileged row — and immediately enters the admin notification + approval set.

src/auth.ts

10 · Behavior gaps

Where the systems disagree with each other

The most dangerous gaps aren't inside one module — they emerge when authorization, navigation, notification, and identity make different assumptions about the same action.

10 · Cross-cutting gaps

Identity drift poisons the whole chain

HIGH Clerk email drift silently drops every notification

Email is the only delivery key. A reviewer who changes their Clerk email stops receiving "ready for review" pings entirely — the request just sits unread in their inbox — because the stale app-DB email matches no one. No log, no fallback.

auth.ts × sender.ts × approvals

HIGH Email-match adoption widens approval + notification scope

Binding a Clerk login to a seeded admin row by unverified email instantly grants approval authority and enrolls that account in the admin recipient set for every in-scope request.

auth.ts × admin controls × notifications

MED Actor self-exclusion rides the drifted email

"Don't notify the actor" is email !== actor.email. If the acting admin's email drifted, they get notified of their own approval — or the wrong admin gets excluded.

submit/coach-decision × auth.ts
10 · Cross-cutting gaps

The review action is hard to reach

MED Coach review is unreachable by nav and mis-linked by notification

/coach/inbox (where coaches actually approve) is in no navbar. The one notification a coach gets deep-links to the requester-framed detail page ("view-only for you"), not to the page where they can act.

navigation.ts × notifications × viewing

MED Admins can act at the coach stage but are never told and can't navigate there

An admin is authorized to coach-approve a SUBMITTED request, but the admin detail route blocks DRAFT/SUBMITTED, and the submit notification goes only to the coach. The only path is guessing the user deep link.

request-access × admin route × notifications

MED Reopen is invisible to notifications, cache, and progress at once

Reopening a rejected request to DRAFT fires nothing, busts no cache, and leaves the progress timeline to re-derive state from a transition that left no trace.

reopen/route.ts × 3 systems

10 · Where to start

Highest-leverage fixes

1 · Add a Clerk webhook (or login re-sync) for name + email

Single fix that closes the notification-drift, actor-exclusion, and display-name gaps at once. Treat email as a synced projection of Clerk, not a copy.

2 · Require verified email before adopting an existing row

Gate the onConflictDoUpdate adoption on Clerk's email_verified to close the seeded-admin takeover vector.

3 · Record policy acceptance on the Clerk path

Stamp policyAcceptedAt/policyVersion at provisioning or first onboarding step; stop hardcoding the false "accepted during registration" copy.

4 · Make reopen a first-class transition

Emit a notification and bust caches like every other transition; add a status pre-check to routes so illegal transitions return 409, not 500.

5 · Tighten viewing & comments

Decide intent for team-member visibility; redact reviewer authorEmail from requester-facing comment surfaces.

6 · Surface the coach inbox & reconcile the docs

Add /coach/inbox to navigation, point submit notifications at it, and re-align user-capabilities.md with the real nav guards.

Summary

The architecture is sound.
The seams need stitching.

RBAC, scoping, and the state machine are well-built and centrally enforced. The risk lives at the boundaries — where Clerk identity, in-app notifications, navigation, and request visibility each assume something slightly different about the same user and the same request.
